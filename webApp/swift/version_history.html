Version 0.6.0

March 5, 2014

4users

  • efeLang challenge set extended: distributed natural language predicates, such as “Akwasi, Jojo and Wizo are each big.”, included. This extension makes it really non-trivial for many players, as appeared to be the case during a preliminary “pencil and paper” experiment among test-players.

  • Better rendering of NL sentences: and comma lists for example. Not “... Pete, Akwasi, Siriwasi.” but “... Pete, Akwasi and Siriwasi.”



4developers

  • [CTLcompare] Comparing answer of computer and player is now done by showing equality between the CTL answers of both (previously: the translations into natural language were compared, but this is problematic, for example orders may be different.) Note: the CTL answer language fort the efeLang-challenge is Fofa (see documentation).

  • Added CTL manipulations (generally applicable, but developed to realise [CTLcompare]):
    - Specifically for the Fofa language (Fofa: see the documentation): equality modulo variable names; bridge based auto translation between Fofa sentences (so, sentences using a different constant name for the same entity can be translated back and forth). Not yet everything is covered, but sufficient for the Efe challenge so far.
    - General (FOL): bridge based term translators added.

Version 0.5.0

February 19, 2014

4users

  • Most important change: a completely new challenge is integrated: efeLang! The previous challenge (!una) turned out too difficult for most beginning players. This challenge will return in subsequent SWiFT versions in a later playing stage.

  • A progression bar is shown, allowing the player to easily jump back to previous rounds for review and/or corrections, and for an enhanced sense of overview for the player. (However, the rounds do not yet automatically rewind when a previous round is re-edited).

4developers

  • The introduction of efeLang is not so easy as it may seem. The previous challenge was still depended on some non-Scala code (Clean). EfeLang is completely written in Scala. Moreover, this has been done in an abstract, generic way. The latter allows other future challenges, with new CTL language and reasoning problems, to be integrated far more easily (at least it is intended to work that way, if it doesn't there is pie.).

  • Moreover, the code is already partially prepared for allowing more than one challenge in the SWiFT game. Considerable refactoring has taken place, among other things, introducing some parameterized types to make it easier to add new challenges with improved reuse.

4developers

  • PersistentDatastructureUpgrader added. This piece of code applies the right upgrades to the persistently stored data, if a server is updated with a new version of SWIFT, but still has data stored in an older format. The upgrade-algorithms belonging to specific versions can simply be added to the list of upgraders, and will be applied when necessary.
